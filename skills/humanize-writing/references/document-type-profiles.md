# Document Type Profiles

What "human-written" means varies by document type. Use these profiles to calibrate eval criteria.

## GitHub README

**Audience:** Developers, hiring managers browsing GitHub profiles
**Expected tone:** Casual-professional. First-person is normal. Humor and personality are welcome. This is NOT a resume — it's closer to a developer's personal blog.
**Human norms:**
- Sentence fragments are fine
- Incomplete thoughts and "work in progress" notes are common
- Project descriptions explain motivation ("I built this because...") not just features
- Tech stack lists are often just badges with no explanation
- Stats cards are standard — don't flag as AI
- Rough edges are expected and authentic

**Common AI tells for this type:**
- Every project described like a product page
- No personality or motivation behind projects
- Overly polished language for a casual platform
- Resume-style achievements instead of project-focused content

## PRD (Product Requirements Document)

**Audience:** Engineers, product managers, stakeholders
**Expected tone:** Professional but direct. Can be opinionated. Should read like a decision-maker wrote it, not a template generator.
**Human norms:**
- Decisions include rationale ("we chose X because...")
- Some sections longer than others (detailed where it matters, brief where obvious)
- May have incomplete sections with TODOs
- Uses domain-specific jargon naturally
- References specific people, teams, or past incidents

**Common AI tells for this type:**
- Every section exactly the same length
- No rationale behind decisions
- Generic acceptance criteria that could apply to any product
- Sanitized language with no opinions

## Blog Post

**Audience:** General readers, industry peers
**Expected tone:** Conversational, opinionated, personal. The author's voice should be distinctive.
**Human norms:**
- Strong opening hook (not a definition or "In today's fast-paced world...")
- Personal anecdotes and experiences
- Opinions stated directly without hedging
- Variable paragraph lengths
- May go on tangents
- Concludes with a personal takeaway, not a generic summary

**Common AI tells for this type:**
- Opens with a broad statement about the industry
- Every paragraph is 3-4 sentences
- Balanced "on one hand / on the other hand" hedging
- Concludes with "In conclusion, [restatement of thesis]"
- No personal voice or anecdotes

## Email / Cover Letter

**Audience:** Specific recipient
**Expected tone:** Professional but warm. Should sound like a real person, not a form letter.
**Human norms:**
- References specific things about the recipient/company
- Typos and imperfect grammar are common (and authentic)
- Variable sentence length
- Personal connection to the role/topic
- Doesn't cover every possible qualification

**Common AI tells for this type:**
- Covers every bullet point from the job description
- Perfect grammar throughout
- Every paragraph starts with "I"
- Generic company compliments ("I'm excited about your innovative approach to...")
- Sign-off with "I look forward to the opportunity to discuss..."

## Documentation

**Audience:** Developers using a tool/library
**Expected tone:** Clear, direct, minimal. Good docs waste no words.
**Human norms:**
- Code examples are central, not decorative
- Explains the "why" alongside the "how"
- Acknowledges limitations and edge cases
- May include warnings from experience ("Note: this will fail silently if...")
- Sections vary dramatically in length based on complexity

**Common AI tells for this type:**
- Every function/method gets the same amount of explanation
- Descriptions restate the function name in sentence form
- No warnings or edge cases mentioned
- Perfect consistency in formatting (real docs evolve over time and have inconsistencies)
